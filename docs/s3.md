# S3 / Object Storage (Long-Term Storage)

TSDB.ai supports tiered long-term storage via any S3-compatible object store. When enabled, the Deduper uploads every canonical block to S3 after writing it locally. The Query Gateway transparently falls back to S3 for any block whose local copy has been evicted.

## Supported Backends

| Backend | Configuration |
|---|---|
| AWS S3 | Leave `endpoint` empty, set `region` + credentials |
| MinIO | Set `endpoint` to MinIO URL, set `use_path_style: true` |
| Cloudflare R2 | Set `endpoint` to `https://<account-id>.r2.cloudflarestorage.com` |
| Any S3-compatible | Set `endpoint` + `use_path_style` as needed |

## Storage Tier Architecture

```
Write Path:
  Ingestor → WAL → Deduper → blocks/canonical/ → S3

Read Path (3-tier):
  Query → LRU Cache → Local Disk → S3
```

Because TSDB.ai's polynomial blocks are extremely compact (typically under 50 KB per 2-minute window regardless of series count), keeping local copies for 30–60 days before evicting to S3 is practical even on modest hardware.

## Enabling S3

Set `s3.enabled: true` in `tsdb.yaml` and provide your bucket and credentials:

```yaml
s3:
  enabled: true
  endpoint: ""                     # leave empty for AWS S3
  region: "us-east-1"
  bucket: "my-tsdb-lts"
  prefix: "blocks/"
  access_key_id: ""                # or set AWS_ACCESS_KEY_ID env var
  secret_access_key: ""            # or set AWS_SECRET_ACCESS_KEY env var
  use_path_style: false
  upload_workers: 4
  upload_queue_capacity: 256
  upload_timeout_s: 60
  download_timeout_s: 30
  retention_after_upload_min: 86400   # keep local copy for 60 days
  multipart_threshold_mb: 100
  multipart_part_size_mb: 50
```

## Full Configuration Reference

| Setting | Default | Description |
|---|---|---|
| `enabled` | false | Master switch |
| `endpoint` | "" | S3-compatible URL (empty = AWS) |
| `region` | "us-east-1" | AWS region for signing |
| `bucket` | "tsdb-ai-lts" | Target bucket (must exist) |
| `prefix` | "blocks/" | Object key prefix |
| `access_key_id` | "" | AWS access key (env var fallback) |
| `secret_access_key` | "" | AWS secret key (env var fallback) |
| `use_path_style` | false | Required for MinIO and some non-AWS endpoints |
| `upload_workers` | 4 | Parallel upload goroutines |
| `upload_queue_capacity` | 256 | Async upload queue depth |
| `upload_timeout_s` | 60 | PutObject HTTP timeout |
| `download_timeout_s` | 30 | GetObject HTTP timeout |
| `retention_after_upload_min` | 86400 | Local retention after S3 upload (60 days) |
| `multipart_threshold_mb` | 100 | Objects larger than this use multipart upload |
| `multipart_part_size_mb` | 50 | Multipart chunk size |

## MinIO Example

```yaml
s3:
  enabled: true
  endpoint: "http://minio:9000"
  region: "us-east-1"
  bucket: "tsdb-blocks"
  prefix: "blocks/"
  access_key_id: "minioadmin"
  secret_access_key: "minioadmin"
  use_path_style: true
```

## Cloudflare R2 Example

```yaml
s3:
  enabled: true
  endpoint: "https://<account-id>.r2.cloudflarestorage.com"
  region: "auto"
  bucket: "tsdb-blocks"
  prefix: "blocks/"
  access_key_id: "<r2-access-key>"
  secret_access_key: "<r2-secret-key>"
  use_path_style: false
```

## Notes

- TSDB.ai does **not** create the bucket — it must exist before starting the service
- Credentials can be supplied via config file or environment variables (`AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`)
- Setting `retention_after_upload_min: 0` keeps local copies forever (S3 acts as a backup replica only)
